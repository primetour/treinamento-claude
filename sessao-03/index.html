De onde viemos

Você já tem os templates. E agora?

Na Sessão 2 você montou prompts com os 4 elementos e salvou seus templates. Mas há um incômodo que ficou.

O incômodo

Toda nova conversa começa do zero. Você recola o tom de voz, reanexa os mesmos arquivos, reexplica quem é o cliente. De novo. E de novo.

O problema

Cada conversa é uma ilha

Lembra da Sessão 1: o Claude começa cada chat do zero e não vê o que houve em outra conversa. Isso tem um custo diário.

Você se repete
O mesmo tom de voz, o mesmo briefing de cliente, os mesmos arquivos — colados de novo a cada conversa.
O contexto se perde
Aquela conversa ótima da semana passada com o cliente VIP some no histórico. Recomeçar é a regra.
O time não compartilha
O bom prompt que um montou fica preso na conversa dele. Ninguém mais aproveita.

Projects existem para resolver exatamente isto: dar um lar permanente ao contexto que se repete.

06 / 31
Conceito

O que é um Project

Um Project é um espaço dentro do claude.ai que guarda instruções fixas e arquivos de conhecimento — e os aplica a todas as conversas criadas ali dentro.

Pense numa pasta com memória: você configura o contexto uma vez, e cada nova conversa já nasce sabendo quem é o cliente, qual o tom, quais os documentos de referência.

Disponível nos planos pagos (Pro e Team). As conversas dentro de um Project ficam agrupadas e separadas do resto.

Em uma frase

“Um colaborador júnior que já leu a pasta do cliente antes de você chamar.”

Em vez de explicar o contexto toda vez, você o entrega uma vez — e ele lembra a cada conversa.

07 / 31
Anatomia

Os três ingredientes de um Project

Todo Project é a soma de três coisas. Você controla as duas primeiras; a terceira se acumula sozinha.

Instruções

O texto fixo que orienta toda conversa: papel, tom de voz, regras, o que sempre considerar. Suas instruções personalizadas.

Arquivos de conhecimento

Os documentos de referência: contratos-modelo, o manual de redação, a ficha do cliente, propostas anteriores.

Histórico de conversas

Tudo o que você já conversou dentro do Project fica reunido ali — fácil de retomar, sem caçar no histórico geral.

08 / 31
Ingrediente 1

Instruções: o briefing permanente

É aqui que entram seus templates da Sessão 2. O que você colava toda vez, agora mora no Project.

O que colocar

  • Papel: quem o Claude deve ser neste contexto.
  • Tom de voz: a voz da PRIMETOUR, do Manual de Redação.
  • Regras fixas: o que sempre fazer, o que nunca fazer.
  • Formato padrão: como você gosta de receber as respostas.
Cuidado

Instrução é permanente: vale para toda conversa do Project. Coloque o que é estável. O que muda a cada caso vai na mensagem, não na instrução.

09 / 31
Ingrediente 2

Conhecimento: a estante de referência

Os arquivos que toda conversa do Project pode consultar, sem você reanexar.

Documentos de marca
Manual de Redação PRIMETOUR, guia de identidade, posicionamento. A voz da casa, sempre à mão.
Material recorrente
Contrato-modelo, tabela de fornecedores, propostas anteriores, fichas de cliente.
Contexto de referência
Relatórios setoriais, briefings de campanha, atas que você consulta com frequência.

Limite prático: o conhecimento do Project tem uma capacidade (mostrada em %). Coloque o que é referência — não o arquivo gigante que você usa uma vez só.

10 / 31
Comparação direta

Conversa avulsa vs Project

A diferença não é de poder — é de repetição. Veja onde cada um se encaixa.

CONVERSA AVULSA

Boa para o pontual

  • Uma pergunta solta, sem continuidade
  • Um assunto que não volta
  • Nada de contexto a preservar
  • Começa rápido, some no histórico
PROJECT

Bom para o recorrente

  • Um cliente, tema ou rotina que volta sempre
  • Tom de voz e regras fixas a manter
  • Arquivos de referência reutilizados
  • Conversas que vale agrupar e retomar
11 / 31
Critério de decisão

Vale criar um Project?

Três perguntas. Se você responde sim a alguma, provavelmente vale.

1
Eu me pego repetindo o mesmo contexto?
Se você cola o mesmo tom, briefing ou arquivo em conversas diferentes, isso pede um Project.
2
Esse assunto vai voltar nas próximas semanas?
Cliente recorrente, rotina de board, linha de produto. Se volta, o contexto merece um lar.
3
Há material de referência que eu sempre consulto?
Manual de Redação, contrato-modelo, tabela de fornecedores — vira conhecimento do Project.

Regra simples: conversa de uma vez só, não crie Project. Contexto que se repete, crie.

12 / 31
Exemplo concreto

Como se parece um bom Project

Um Project de comunicação da PRIMETOUR, montado para nunca mais reexplicar a voz da marca.

Project · Comunicação PRIMETOUR

Instruções: papel de redator sênior da casa + a voz editorial do Manual de Redação + regras duras (PRIMETOUR em caixa alta, “traslado” nunca “transfer”, sem termos banidos).

Conhecimento: Manual de Redação PRIMETOUR, exemplos de textos aprovados.

Uso: toda conversa já escreve na voz certa, sem briefing.

Por que funciona

  • O tom de voz está no Project — não num prompt que você cola.
  • O Manual vive como conhecimento — o Claude consulta direto.
  • Qualquer texto (e-mail, post, comunicado) já sai alinhado.
  • Quando o Manual muda, você atualiza num lugar só.

14 / 31
Honestidade

O que não colocar num Project

Um Project bom é curado. Encher de tudo atrapalha tanto quanto deixar vazio.

Dado pessoal sensível sem necessidade
CPF, passaporte, cartão, dado de saúde do cliente. A política completa vem na Sessão 6 — por ora, na dúvida, não coloque.
Material que muda toda hora
Tabela de preços que vira semana, agenda do dia. Instrução é para o estável; o volátil vai na conversa.
Arquivos enormes usados uma vez
O relatório de 300 páginas que você analisa num único dia. Anexe na conversa, não no conhecimento permanente.

16 / 31
Manutenção

Quatro hábitos de quem usa bem

1
Nome claro e específico
“Cliente — Família Almeida”, não “Cliente 1”. Você vai ter vários; o nome é como você acha.
2
Instruções enxutas
Meia página bem escrita vale mais que três páginas. O essencial e estável, não tudo.
3
Revisão periódica
Uma vez por mês, abra e pergunte: a instrução ainda está certa? O arquivo ainda é o atual?
4
Comece com um, não dez
Monte um Project, use uma semana, sinta o valor. Depois multiplique. Não crie dez vazios hoje.
17 / 31
Honestidade

Onde Projects não fazem mágica

Projects organizam contexto. Não consertam o que já vimos que a IA não faz.

Não eliminam a alucinação
Mesmo com bons arquivos, valide o factual crítico. O conhecimento ajuda, não isenta a conferência.
Não substituem o bom prompt
O Project dá o contexto; a tarefa clara ainda é sua, conversa a conversa. A Sessão 2 continua valendo.
Não são um banco de dados vivo
O conhecimento é uma foto do que você subiu. Mudou o arquivo? Atualize no Project — ele não busca sozinho.

O lab

Do contexto solto ao Project que trabalha por você

Escolha um contexto recorrente do seu dia. Monte o Project. Teste com uma conversa real. Em 45 minutos.

Enunciado

Crie um Project para um cliente, tema ou rotina que se repete no seu trabalho. Escreva as instruções, anexe um arquivo de referência (pode ser o Manual de Redação) e abra uma conversa dentro dele para confirmar que o contexto está valendo.

Critério de sucesso

Ao final, abra uma conversa nova no Project e faça um pedido sem dar nenhum contexto. Se a resposta já vier com o tom e a referência certos, o Project está funcionando.

Instruções · S3

01 — Instruções base de um Project

Quando usar: O esqueleto de qualquer Project. Cole no campo de instruções e preencha.

PAPEL: Você é [o especialista ideal para este contexto, ex.: redator sênior da PRIMETOUR].

CONTEXTO PERMANENTE: [o que é sempre verdade aqui — cliente, tema, objetivo recorrente].

COMO RESPONDER:
- Tom: [a voz desejada].
- Formato padrão: [como gosto de receber — ex.: direto, em tópicos, máximo 1 página].
- Sempre considere: [regras fixas deste contexto].

NUNCA: [o que evitar sempre].

Quando faltar uma informação para responder bem, pergunte antes de assumir.

Dica: Mantenha enxuto. Instrução é para o que é estável; o que muda a cada caso vai na mensagem da conversa.

Instruções · S3

02 — Project “Comunicação PRIMETOUR”

Quando usar: Um Project para todo texto da marca sair na voz certa. Some o Manual de Redação como conhecimento.

PAPEL: Você é redator sênior da PRIMETOUR e conhece o Manual de Redação anexado.

VOZ: luxo que não grita — elegância contida, quiet luxury. Curadoria que conversa com um viajante exigente. Autoridade leve. Venda a experiência (o "como"), não só o destino.

REGRAS DURAS:
- A marca é sempre PRIMETOUR, em caixa alta.
- "para", nunca "pra". "traslado", nunca "transfer". Sem diminutivos.
- Termos proibidos: luxuoso, refúgio, escapada, paraíso, idílico, de tirar o fôlego, explorar, desbravar, instagramável.

FORMATO: traduza abstrações em cenas sensoriais; adjetivação controlada; alterne frases curtas de impacto com períodos de imersão.

Antes de finalizar qualquer texto, revise-o contra estas regras.

Dica: Anexe o arquivo do Manual de Redação ao conhecimento do Project. Assim o Claude consulta a fonte, não só o resumo acima.

Instruções · S3

03 — Project “Cliente VIP [Nome]”

Quando usar: Um cliente de alto valor recorrente. Cada conversa já conhece a relação.

PAPEL: Você é o gerente de relacionamento dedicado ao cliente [Nome/codinome].

QUEM É O CLIENTE: [perfil, há quanto tempo viaja conosco, peso, estilo de viagem].
PREFERÊNCIAS: [destinos, ritmo, hospedagem, restrições, o que valoriza].
HISTÓRICO RELEVANTE: [viagens recentes, elogios, atritos a evitar].

COMO RESPONDER:
- Tom: caloroso, atencioso, à altura de um cliente exigente.
- Sempre proteja a relação de longo prazo.
- Antecipe o próximo passo: o que oferecer, o que confirmar.

NUNCA exponha dado pessoal sensível em textos que saem para fora.

Dica: Use codinome em vez do nome real enquanto a política de dados (Sessão 6) não estiver fechada. O contexto funciona igual.

Instruções · S3

04 — Project “Propostas & Fornecedores”

Quando usar: Analisar propostas, contratos e atas com a mesma régua, sempre.

PAPEL: Você é analista sênior de contratos e propostas da PRIMETOUR.

CONTEXTO: avalio propostas de fornecedores de hospedagem, traslados e experiências. Tenho um contrato-modelo e uma tabela de referência anexados.

AO ANALISAR QUALQUER PROPOSTA, SEMPRE DEVOLVA:
1. Resumo em 5 linhas
2. As 3 cláusulas ou pontos de maior risco para nós
3. Comparação com nosso padrão de referência (o que foge)
4. Uma pergunta a devolver ao fornecedor antes de fechar

Tom direto, sem rodeios. Aponte o risco mesmo quando a proposta parecer boa.

Dica: Anexe o contrato-modelo e a tabela ao conhecimento. Aí cada nova proposta é comparada com o padrão automaticamente.

Instruções · S3

05 — Project “Board & Reuniões”

Quando usar: Preparar reuniões de board e encontros decisivos com contexto recorrente.

PAPEL: Você é meu chefe de gabinete, preparando reuniões decisivas.

CONTEXTO: participo do board da PRIMETOUR. As pautas recorrentes envolvem [temas]. O conselho costuma pesar [critérios].

AO PREPARAR QUALQUER REUNIÃO, ME ENTREGUE:
1. Síntese do que está em jogo (5 linhas)
2. As 3 perguntas que o conselho fará primeiro
3. Duas objeções prováveis e resposta curta para cada
4. Um único ponto que eu não posso esquecer de dizer

Tom executivo. Antecipe o difícil, não o confortável.

Dica: Combine com o template 07 da Sessão 1 (antecipação de objeções) dentro deste Project para preparar reuniões específicas.

Montagem · S3

06 — O Claude escreve as instruções do seu Project

Quando usar: Você sabe o contexto, mas não sabe como redigir as instruções. Use numa conversa avulsa.

Quero criar um Project no Claude para o seguinte contexto recorrente:

CONTEXTO: [descreva: cliente, tema ou rotina; o que se repete]
O QUE EU SEMPRE PRECISO: [tipo de entregável, tom, regras]
ARQUIVOS QUE TENHO PARA ANEXAR: [liste]

Escreva as instruções ideais para esse Project — papel, contexto permanente, como responder e o que evitar. Deixe enxuto e pronto para colar. Ao final, sugira que arquivos vale anexar como conhecimento.

Dica: Compare o que ele escreveu com o template 01. Ajuste e cole no campo de instruções do Project.

Montagem · S3

07 — Checklist do que anexar como conhecimento

Quando usar: Na dúvida sobre quais arquivos colocar no conhecimento do Project.

Vou montar um Project sobre [contexto]. Tenho estes arquivos candidatos a conhecimento:

[liste os arquivos, um por linha]

Para cada um, diga:
- Vale anexar como conhecimento permanente, ou é melhor usar só na conversa quando precisar? (e por quê)
- Há algum que contém dado sensível e deveria ficar de fora por ora?

Ao final, liste o que falta — algum arquivo de referência que um Project assim deveria ter e eu não listei.

Dica: Conhecimento é curadoria. O arquivo que você usa uma vez vai na conversa; o que consulta sempre vai no Project.

Manutenção · S3

08 — Auditar e melhorar um Project

Quando usar: Um Project não está rendendo o esperado e você quer saber por quê.

Vou colar as instruções atuais de um Project meu. Audite-as.

[cole as instruções do Project]

O QUE EU ESPERAVA DELE: [descreva]
ONDE ELE FALHA HOJE: [descreva, se souber]

Diga:
1. O que nas instruções está vago, conflitante ou faltando
2. O que está sobrando (instrução que não muda nada na prática)
3. Uma versão reescrita, mais enxuta e clara
4. Se o problema parece ser de instrução ou de arquivo de conhecimento

Dica: Rode esta auditoria de tempos em tempos. Projects acumulam instruções zumbis com o uso.

Manutenção · S3

09 — Manutenção mensal do Project

Quando usar: Uma revisão rápida e periódica para o Project não envelhecer.

Vou colar as instruções e a lista de arquivos de conhecimento de um Project meu. Faça uma revisão de manutenção.

INSTRUÇÕES ATUAIS:
[cole]

ARQUIVOS DE CONHECIMENTO:
[liste, com a data aproximada de cada]

Me ajude a decidir:
1. Alguma instrução ficou desatualizada e precisa mudar?
2. Algum arquivo virou versão antiga e deveria ser substituído?
3. Há algo que eu venho colando nas conversas e que deveria virar instrução fixa?

Dica: Marque um lembrete mensal. Cinco minutos de manutenção evitam um Project que responde com base em informação velha.

Decisão · S3

10 — Vale criar um Project para isto?

Quando usar: Você está na dúvida entre abrir um Project ou só uma conversa avulsa.

Estou decidindo se crio um Project para a seguinte situação:

SITUAÇÃO: [descreva o que você precisa fazer com o Claude]
FREQUÊNCIA: [é uma vez só, ou volta? com que regularidade?]
CONTEXTO ENVOLVIDO: [tom, arquivos, briefing que se repetiria]

Me diga, de forma direta:
- Isto justifica um Project, ou uma conversa avulsa resolve melhor? Por quê.
- Se justifica, como eu nomearia e estruturaria (por cliente, tema ou cadeira)?
- Se não justifica agora, o que precisaria mudar para passar a justificar.

Dica: Na dúvida, comece com uma conversa avulsa. Quando perceber que está repetindo o contexto, aí promova para Project.

Glossário

Termos que apareceram nesta sessão

Project (Claude)
Espaço no claude.ai que guarda instruções fixas e arquivos de conhecimento, aplicados a todas as conversas criadas ali dentro. Disponível nos planos pagos.
Instruções do Project
O texto fixo que orienta toda conversa do Project: papel, tom de voz, regras e formato padrão. O que você antes colava a cada conversa.
Conhecimento (knowledge)
Os arquivos de referência anexados ao Project que toda conversa pode consultar sem reanexar. Tem capacidade limitada — é curadoria, não depósito.
Contexto recorrente
A informação que se repete entre conversas (cliente, tom, documentos). É o que justifica criar um Project em vez de uma conversa avulsa.
Conversa avulsa
Um chat fora de qualquer Project. Bom para o pontual; não preserva contexto entre uma conversa e outra.
Template de prompt
Um prompt com lacunas, reusável. Na Sessão 3, deixa de ser colado e passa a viver como instrução de um Project.
